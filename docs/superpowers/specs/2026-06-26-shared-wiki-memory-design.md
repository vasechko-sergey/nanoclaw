# Общая вики-память для агентов — дизайн

- **Дата:** 2026-06-26
- **Статус:** утверждён к планированию
- **Агенты:** jarvis, greg, gordon, payne, scrooge (все 5, per-person)

## Проблема

Сейчас знание распределено неравномерно:

- У **jarvis** — полноценная wiki (`memories/`: люди / проекты / встречи) + субагенты `wiki-ingest` / `wiki-lint`, схема зашита в его `CLAUDE.md`. Механизм не переиспользуется.
- У **greg / gordon / payne / scrooge** — только `memories/` (ленивый каталог `index.md`), без ingest-пайплайна и без дисциплины.
- **Общая память** = `/workspace/global` (per-person): `identity.md` + `about.md`, RW только у одного писателя (`.writer` = jarvis), RO остальным. Это конфиг идентичности владельца, не база знаний.

Каждому агенту нужна своя предметная вики (Greg — медстатьи, Gordon — питание, Payne — тренировки, Scrooge — финансы), плюс общий раздел, куда все могут писать выводы и откуда все читают. Механизм — единый.

## Решения (зафиксированы в брейнсторме)

1. **Топология:** приватная wiki у каждого из 5 агентов + общая зона на запись для всех. Приват не засоряет контекст других; общее полезно для кросс-агентных фактов.
2. **Модель записи в общее:** все маунтят общую RW, разделение по доменам (один блок — один агент-владелец, soft-конвенция, не FS-права). Без спорного мастер-индекса; общий `log.md` — append-only.
3. **Механизм:** единый, описан в общем `groups/INSTRUCTIONS.md` (а не в отдельном container-скиле и не дублируется по CLAUDE.md). Каждый `CLAUDE.md` ссылается на инструкцию + называет свой блок.
4. **Граница приват/общее:** приват = сырьё + глубокий разбор; общее = компактные action-выводы.
5. **Единый принцип (триада):** ingest и lint работают **только с приватной** памятью агента; в общее знание попадает через отдельный осознанный **publish** из готового приватного результата — тот же принцип, что уже работает для `public.md` → `profiles/`.

## Архитектура

### Двухуровневая модель памяти

Каждый агент получает два различимых хранилища:

**Приватная wiki** — существующий `memories/` агента + новый под-каталог `memories/sources/` (архив сырья). Хранит полные источники (статьи, PDF, транскрипты), глубокий разбор, черновики, доменные заметки. Читает только сам агент → не засоряет контекст остальных. Уже смонтирован RW в составе `/workspace/agent/` (per-person, `userMemoryRoot`).

**Общая зона** — новый per-person каталог, смонтированный RW для **всех 5** агентов в `/workspace/shared/`. Хранит компактные action-выводы, размеченные по доменам. Все читают всё; каждый пишет только в свой блок (через publish).

**Не трогаем, отдельные поверхности:**

- `/workspace/global/` (`identity.md` + `about.md`, single-writer jarvis) — конфиг идентичности владельца. Без изменений.
- `profiles/` + publish — суточный статус-снапшот для утреннего брифа. Без изменений.
- Приватная CRM jarvis (`memories/people|projects|interactions`) — остаётся приватной.

Общая вики — третья, отдельная поверхность: долгоживущее накопленное знание.

### Структура общей зоны

```
/workspace/shared/        (= data/user-memory/<person>/shared/, RW для всех 5)
  README.md               статичная карта блоков (правится редко, вручную)
  log.md                  append-only; 1 строка на каждый publish, любой агент
  nutrition/   ← gordon    в каждом блоке: index.md (пишет владелец блока) + страницы-выводы
  training/    ← payne
  health/      ← greg
  finance/     ← scrooge
  general/     ← jarvis    кросс-доменное знание о владельце/жизни (не identity-baseline)
```

Конвенции:

- **Нет мастер-индекса** → убирает спорный файл-клоббер. У каждого блока свой `index.md`, пишет его только владелец блока.
- **`log.md`** — единственный мульти-райтер файл → append-only, 1 строка на запись: `## [YYYY-MM-DD] <agent> <domain> | <что опубликовано>`. Худшая коллизия append = потеря одной строки, не порча файла. Приемлемо.
- **Soft-владение:** «пиши только в свой блок» — конвенция из INSTRUCTIONS + CLAUDE.md, не FS-права (всё RW). Чтение не ограничено. Блоки заводятся по мере появления доменов, не жёстко.
- `general/` (jarvis) ≠ `about.md`: `about.md` — baseline идентичности (кто владелец), `general/` — накопленные кросс-доменные выводы о жизни/логистике. Лёгкое пересечение допустимо.

## Механизм — три операции, единые для всех (в `INSTRUCTIONS.md §Wiki`)

Новая секция в общем `groups/INSTRUCTIONS.md`. Единственный источник истины, читается всеми 5 через `@./INSTRUCTIONS.md`. Описывает триаду:

**Ingest** (через субагент, изоляция контекста) → пишет **только в приват**. Сырьё → `memories/sources/`, глубокий разбор → страницы `memories/`. Длинный источник не попадает в основной контекст. Дисциплина: один источник за раз, полностью завершить (страницы + `memories/index.md` + лог приватной wiki) перед следующим.

**Lint** (через субагент) → работает с **собственной приватной** wiki агента. Health-check: orphan-страницы, противоречия, устаревшее, пробелы. Каждый агент линтит сам себя. Общую зону **не трогает**.

**Publish** (основной агент, осознанно) → из готового приватного результата выжимает компактный action-вывод и пишет в **свой** блок `/workspace/shared/<domain>/` + добавляет строку в `log.md`. Единственный путь в общее. Тот же принцип и дисциплина, что у существующего `public.md` → `profiles/`.

Следствие: общая зона = только курированные опубликованные выводы — никогда сырьё, никогда случайно. Каждая запись в общее намеренная и в своём блоке → риск клоббера около нуля (мульти-райтер только append в `log.md`).

«Publish» становится одним глаголом с двумя целями: `profiles/` (суточный статус, существует) + `shared/<domain>/` (долговечные выводы, новое). Приватная wiki — где живут ingest+lint; общая — только publish.

### Вызов субагентов

Ingest/lint вызываются **инлайн-промптом** через `Task`, без файлов `AGENT.md`:

```
Task({
  description: "wiki ingest",
  prompt: "Прочти /workspace/agent/INSTRUCTIONS.md §Wiki (раздел ingest) и обработай источник: <source>. Работай только с приватной памятью."
})
```

Без файлов-субагентов, ноль дублирования. Существующие `groups/jarvis/agents/wiki-ingest/AGENT.md` и `.../wiki-lint/AGENT.md` ретайрятся (дисциплина теперь в INSTRUCTIONS).

## Деплой

- **Код хоста** (`user-memory.ts`, `container-runner.ts`): commit в main → `pnpm run build` → рестарт. Локально и на VDS (git pull + build + рестарт).
- **`groups/INSTRUCTIONS.md`** (gitignored, канон на VDS): правка локально → **scp на VDS** → rebirth живых сессий (kill контейнер + DELETE continuation-row по всем сессиям агента, `find` не glob), иначе живой агент держит старые инструкции (SDK resume).
- **Per-agent `CLAUDE.md`** (groups/ gitignored): scp на VDS + rebirth, как INSTRUCTIONS.
- **Общая зона** (каталоги `shared/`): хост авто-скаффолдит на следующем спавне.

## Тесты / верификация

- **Юнит** (vitest, host): `userSharedRoot` возвращает верный путь; `initUserMemory` идемпотентно создаёт `shared/` + блоки + `log.md` + `README.md`; повторный вызов не падает и не затирает существующее.
- **Юнит:** маунт `/workspace/shared` присутствует с `readonly:false` для произвольного агента (в отличие от global, который RO для не-writer).
- **e2e (ручной, Сергей):**
  1. Greg: ingest медстатьи → сырьё в `memories/sources/`, разбор в `memories/`, общее пусто.
  2. Greg: publish вывода → строка в `shared/health/`, запись в `shared/log.md`.
  3. Gordon: читает `shared/health/` (кросс-чтение), пишет в `shared/nutrition/`, не трогает `health/`.
  4. Lint Greg → работает с приватом Greg, общую зону не меняет.

## Не-цели (YAGNI)

- Жёсткий FS-контроль доступа к блокам (per-domain права записи). Soft-конвенция достаточна для личной системы из 5 агентов.
- Мастер-индекс общей зоны / поиск (qmd и т.п.). Per-block `index.md` хватает на текущем масштабе.
- Перенос `about.md` / `identity.md` в общую вики. Остаются отдельным конфигом идентичности.
- Слияние существующего `publish`-скила со shared-publish. Только выравнивание принципа в INSTRUCTIONS.

## Риски

- **Миграция jarvis** трогает рабочего агента → последним шагом, с проверкой; приватная CRM не затрагивается.
- **Конкурентная запись:** только `log.md` мульти-райтер; append-only + доменное разделение страниц (один владелец на файл) сводят риск к потере максимум одной лог-строки.
- **Контекст-стоимость INSTRUCTIONS:** секция §Wiki всегда в контексте всех сессий. Держать концизной (~40–60 строк); тяжёлая ingest-дисциплина — кратко, детали отрабатывает субагент.
