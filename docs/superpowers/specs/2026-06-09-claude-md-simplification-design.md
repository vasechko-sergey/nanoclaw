# CLAUDE.md Simplification Design

**Date:** 2026-06-09
**Status:** Design
**Author:** Sergei + Jarvis

## Problem

Operator's per-agent system prompt today is assembled from four places:

1. `container/CLAUDE.md` — shared base, mounted into every agent container.
2. `groups/<folder>/CLAUDE.md` — either a "composed" file with `@`-imports only or a manual persona file. The composed branch is regenerated on every spawn by `composeGroupClaudeMd`.
3. `groups/<folder>/CLAUDE.local.md` — per-group memory, auto-loaded by Claude Code.
4. `groups/<folder>/.claude-fragments/<skill>.md` and `<module>.md` — symlinks/inlines pulled from `container/skills/*/instructions.md` and `container/agent-runner/src/mcp-tools/*.instructions.md`, plus inline MCP server instructions from `container.json`.

In practice this is opaque ("what's actually in the prompt?") and forces three different override semantics: shared can be overridden by group CLAUDE.md, fragments can be overridden by import order, and CLAUDE.local.md sits between them as auto-loaded memory. Earlier today this caused real damage: `container/CLAUDE.md` prescribed "be concise" and silently flattened Greg's House persona and Payne's drill-instructor voice. Fixing it required adding an explicit "persona overrides the default register" line — papering over an architectural mistake rather than removing it.

The operator wants **one file per agent for everything that's "this agent"**, plus a single shared mechanism for everything that's truly shared, plus a clean separation for memory.

## Goals

- Open `groups/<agent>/CLAUDE.md` and see (or trivially follow) the entire prompt the agent loads.
- One owner per file: persona-text is operator-edited, memory is agent-written, shared rules are host-generated.
- No "composed entry point" with implicit imports — at most one explicit `@`-line.
- Drop the override semantics. If two files conflict, the rule is unambiguous (persona always wins).
- Keep `container/skills/` and `container/agent-runner/src/mcp-tools/` as the source of truth for skill/MCP instructions — collect them into one shared file rather than fragmenting each into the per-agent dir.

## Non-Goals

- Replacing markdown memory with a structured store (key-value, vector DB). Out of scope.
- Touching the runtime addendum (`buildSystemPromptAddendum`) — name + destinations stay where they are.
- Changing how Claude Code SDK auto-loads `CLAUDE.md` from cwd or how `@`-imports resolve. We work within those.

## Design

### File layout (after refactor)

```
groups/
  INSTRUCTIONS.md            # host-regenerated; shared workspace + memory + skill/MCP rules
  <agent>/
    CLAUDE.md                # operator-owned: persona; first line is `@../INSTRUCTIONS.md`
    memories/
      index.md               # agent-maintained: catalog of what lives in memories/
      ...                    # agent's choice — subdirs, files, whatever fits
```

That's the whole file inventory. **Two files per agent** (`CLAUDE.md`, `memories/index.md`) plus the shared `INSTRUCTIONS.md`. No `CLAUDE.local.md`, no `.claude-shared.md`, no `.claude-fragments/`, no `container/CLAUDE.md` as a separate concept.

### `groups/INSTRUCTIONS.md` — generated, single source of shared rules

Generated on every host start (and whenever skill / MCP-server inventory changes). Composed deterministically from:

- A small handwritten preamble baked into the generator: workspace rules, the memory contract.
- All `container/skills/*/instructions.md` (one section per skill).
- All `container/agent-runner/src/mcp-tools/*.instructions.md` (one section per MCP tool module).

Layout:

```
# NanoClaw Agent Instructions

> Generated by the host. Do not hand-edit — your changes will be overwritten.

## Workspace
...

## Memory
Memory lives in `memories/` next to your CLAUDE.md. `memories/index.md` is the
required catalog — keep it in sync. Format inside `memories/` is yours; describe
it in your CLAUDE.md.

Read-loop: when a person/project/event is mentioned, silently check `index.md`
first; read the matching file before answering. Never announce "I have no
record"; just act.

Write-loop: a new fact updates the relevant file immediately, without
confirmation. Add a line to `index.md` when you add a file.

## Skills
### welcome
<content of container/skills/welcome/instructions.md>

### slack-formatting
<content of container/skills/slack-formatting/instructions.md>

...

## MCP tools
### workout
<content of container/agent-runner/src/mcp-tools/workout.instructions.md>

### scheduling
...
```

The file lives at `groups/INSTRUCTIONS.md` so every agent dir at `groups/<agent>/` references it as `@../INSTRUCTIONS.md`.

### `groups/<agent>/CLAUDE.md` — operator-owned, one shape

```
@../INSTRUCTIONS.md

# <Agent name>

<persona content — voice, tone, rules, behaviour. Whatever the operator wants.>

## Memory layout
<this agent's choice of how it organizes memories/. e.g. "self/, people/,
projects/, interactions/<YYYY-MM>/" for Jarvis; "state.md +
trends/<YYYY-WW>.md" for Greg; "programs/, sessions/<date>.json, INDEX.md"
for Payne.>
```

Operator never sees `.claude-shared.md`, no fragments, no composed header. The first line is the only host-coupled bit, and even that is just one explicit import the operator can read.

### `groups/<agent>/memories/index.md` — agent-maintained

Required. Format example:

```
# Memory index

## People
- [people/ivan.md] — work, ABC, last contact 2026-05-30
- [people/anna.md] — friend, mentioned 2026-05-12

## Self
- [self/profile.md] — baseline preferences, location, calendar
- [self/health.md] — weekly trend aggregate (HRV, RHR, sleep)

## Projects
(none yet)
```

The agent rewrites this whenever it adds or renames a memory file. The rule lives in the shared `## Memory` section of `INSTRUCTIONS.md`.

### Removed artifacts

| Artifact | Where its content moves |
|---|---|
| `container/CLAUDE.md` | Preamble of `INSTRUCTIONS.md`. |
| `groups/<agent>/.claude-shared.md` (symlink) | Deleted. Replaced by `@../INSTRUCTIONS.md` in CLAUDE.md. |
| `groups/<agent>/.claude-fragments/` | Deleted. Skill + MCP instructions go into `INSTRUCTIONS.md` sections. |
| `groups/<agent>/CLAUDE.local.md` | If it had any content the operator wrote, the operator copies it into `CLAUDE.md`. Anything written by past agent turns moves to `memories/` (we'll do the migration; details below). |

### `composeGroupClaudeMd` becomes `regenerateSharedInstructions`

Rename + simplify in `src/claude-md-compose.ts`:

- Drop `composeGroupClaudeMd`. Drop the COMPOSED_HEADER detection, the manual-vs-composed branch, fragments-dir reconciliation, all of it.
- Drop `migrateGroupsToClaudeLocal` after the new migration runs once (deletion deferred until any active install has migrated).
- New `regenerateSharedInstructions()` builds `groups/INSTRUCTIONS.md` from the preamble + skill files + MCP module files. Idempotent. Atomic write. Logs nothing on the no-change path.
- Called from `container-runner.buildMounts()` (same site as today's compose). Also called from `src/index.ts` on startup so the file exists on a fresh boot even before the first container spawn.

### Where INSTRUCTIONS.md actually loads from

Claude Code SDK auto-loads `CLAUDE.md` from cwd and follows `@`-imports relative to the file. The CLAUDE.md sits at `/workspace/agent/CLAUDE.md` inside the container (`groups/<agent>/` is mounted at `/workspace/agent`). For `@../INSTRUCTIONS.md` to resolve, `groups/` itself must be mounted at `/workspace/` (parent of `/workspace/agent`). Two ways to make this work:

- **A — mount `groups/` as `/workspace`**, with the per-agent dir as `/workspace/agent`. Requires verifying current mount layout; if today only `groups/<agent>` is mounted, we extend the mount to `groups/` read-only with the per-agent dir read-write inside it. Likely the cleanest path.
- **B — symlink `INSTRUCTIONS.md` into each agent dir** at `groups/<agent>/INSTRUCTIONS.md` (target: `../INSTRUCTIONS.md`). Then CLAUDE.md imports `@./INSTRUCTIONS.md`. No mount changes; one symlink per agent maintained by `regenerateSharedInstructions`.

Pick **B** — symlinks. The per-agent dir already has the right mount semantics; adding one symlink is cheaper than reworking the mount tree and risking unrelated breakage. The operator still sees one explicit import in CLAUDE.md (`@./INSTRUCTIONS.md` — same shape) and one symlink in the dir listing.

### Memory contract (in shared INSTRUCTIONS.md)

Single source of memory rules, replacing today's `container/CLAUDE.md` "Memory" + "Workspace" sections:

```
## Memory

Your memory is the `memories/` folder next to this CLAUDE.md.

Required:
- `memories/index.md` — catalog of what lives in memories/. Update it whenever
  you create or rename a memory file.

Otherwise free-form. Your CLAUDE.md describes your own structure (e.g. wiki,
state files, structured JSON sessions). Pick what fits the work.

Read before answering:
- A person, project, or past event is mentioned → check `memories/index.md`
  silently. If the index points to a relevant file, read it. Weave the result
  into your reply; never announce "I have no record."

Write on new fact:
- Operator shares something substantive → file it now. No confirmation.
- New file → add a one-line entry to `memories/index.md`.
- Renamed / moved file → update the index in the same edit.
```

This is THE contract. Nothing in per-agent CLAUDE.md is allowed to weaken it; per-agent CLAUDE.md only adds structure.

## Migration (one-shot)

`src/migrate-claude-md-v2.ts`, run on startup once (guarded by a `schema_version` row or sentinel file). Steps per `groups/<folder>/`:

1. If `CLAUDE.local.md` exists and non-empty: move its content into `memories/CLAUDE.local-archive.md`, add an `index.md` entry pointing at it. Operator decides later whether to fold into wiki / delete.
2. If `CLAUDE.md` starts with the old `COMPOSED_HEADER`: replace it with a one-line `@./INSTRUCTIONS.md` followed by a placeholder persona stub (`# <agent name>\n\nTODO: persona`). Operator fills in.
3. If `CLAUDE.md` is manual (no header): prepend `@./INSTRUCTIONS.md` as the first line, leave the rest untouched.
4. If `memories/` exists but `memories/index.md` doesn't: create a minimal one listing every file in `memories/`.
5. Delete `.claude-shared.md` and `.claude-fragments/`. Idempotent.

Then `regenerateSharedInstructions()` writes `groups/INSTRUCTIONS.md` and the per-agent symlinks.

Migration writes a `groups/.migrated-claude-md-v2` sentinel file. Subsequent boots see it and skip.

## Error handling

- Skill or MCP module's `*.instructions.md` is missing or unreadable: skip it, log warn with the path. Other sections still land. Failure to read one fragment never blocks generation.
- Operator hand-edits `INSTRUCTIONS.md`: regeneration overwrites silently. Pre-write check: if `INSTRUCTIONS.md` doesn't start with the generator's marker line (`> Generated by the host…`), refuse to overwrite and log error — defensive escape hatch in case someone customizes without realizing.
- `memories/index.md` missing at runtime: agent is told (in the shared rules) it MUST exist. If absent, the agent creates a minimal one on first need. No host-side enforcement; this is an agent-level contract.

## Testing

- `src/claude-md-compose.test.ts` (extend or replace with `instructions-gen.test.ts`):
  - Generates `INSTRUCTIONS.md` from a temp `container/skills/` + `mcp-tools/` tree. Output has the preamble, the right section count, deterministic ordering.
  - Idempotent: second call with no input changes leaves mtime stable (or content byte-identical at least).
  - Refuses to overwrite a non-generator-marker file.
- `src/migrate-claude-md-v2.test.ts`:
  - Composed CLAUDE.md → rewrites correctly.
  - Manual CLAUDE.md → prepends import line, preserves rest.
  - `CLAUDE.local.md` non-empty → archived into `memories/`.
  - Sentinel file prevents re-run.
- E2E smoke: existing dispatcher / bootstrap-trio / adapter-route tests stay green. Host startup with the new path produces a valid `INSTRUCTIONS.md` and three valid CLAUDE.md files for the trio.

## Out of scope

- Replacing per-skill `instructions.md` source-of-truth files (they remain where they are).
- Changing how the operator writes a new skill (still `container/skills/<name>/instructions.md`).
- Removing the runtime addendum (`buildSystemPromptAddendum`) — name + destinations + persona reminder stay.
- Memory tooling (`memory.recall` MCP, vector store, etc.) — separate spec if/when needed.

## Open Questions

None blocking. The mount-vs-symlink decision (B) can be revisited if symlinks turn out to behave badly with Claude Code's @-import resolver inside the container.
